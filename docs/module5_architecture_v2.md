# Module 5 — Redesign Specification Engine
### Architecture & Design Document (Final — v2, deterministic)

**Repository:** `poison-2-0-0-7/thumbnail-ai`
**Baseline commit reviewed:** `95f665e` — "Complete Module 4 thumbnail intelligence pipeline"
**Status of this document:** design only. No implementation or placeholder code. This version supersedes the v1 draft (which proposed a second Ollama call) following architectural review — see §0.

---

## 0. What changed since v1, and why

The v1 draft proposed a second Ollama reasoning stage in Module 5 to compile `ThumbnailIntelligence` into `RedesignSpecification`. On review, that call is being **removed**. Reasoning, in brief (full argument in the prior turn):

- The large majority of `RedesignSpecification`'s fields are directly derivable from numbers `ThumbnailIntelligence` already contains (`composition.clutter_score`, `colors.brightness`, `faces.faces[i].bbox`, etc.) — no interpretation is needed, only threshold rules and arithmetic.
- The two fields that genuinely resist deterministic derivation — a specific new hex palette and invented on-thumbnail copy — are **removed from Module 5's scope entirely** rather than generated by a second LLM call. They're copywriting/creative-generation concerns, not "structurally describe what should change" concerns.
- The repo already contains concrete evidence that a second, larger LLM call on this pipeline is a reliability risk: `ollama_response.txt`, one of the uploaded artifacts, is qwen3:8b producing `"this is not json at all"` on Module 4's *existing, smaller* schema.
- This also restores the line the codebase already draws for itself: Modules 1–3 and five of Module 4's six stages are deterministic; Ollama is reached for exactly once, for the one genuinely irreducible judgment (CTR/curiosity potential). Module 5, as originally scoped, broke that principle. This version restores it.

**Net effect: Module 4 remains the only AI reasoning module in the pipeline. Module 5 is pure, deterministic Python with zero network calls.**

## 2. Module 5's job, precisely

Module 5 converts `ThumbnailIntelligence` — Module 4's complete report, which already contains every CV measurement and the one piece of AI judgment the pipeline needs — into a fully structured `RedesignSpecification`, using only **arithmetic, threshold comparisons, and rule tables** applied to fields that already exist on the input.

Module 5 does **not**:
- Touch pixels. It never opens `thumbnail_path`.
- Re-run OCR, face detection, object detection, color analysis, or composition analysis.
- Generate, edit, or preview any image.
- **Make any network call, of any kind.** No Ollama, no HTTP, no external service. This is now a hard structural property, not just a documented intent: the module has no `requests` import and no retry/tenacity dependency.
- **Invent new creative content.** No new hex colors beyond a deterministic transform of existing ones, no new on-thumbnail copy. Both are explicitly out of scope (see §4.1).

### 2.1 Why deterministic is sufficient here

Every quantitative target in `RedesignSpecification` maps onto a quantitative field Module 4 already computed:

| Target field | Deterministic derivation |
|---|---|
| `layout_direction.target_clutter_score` | Rule: if `composition.clutter_score > CLUTTER_HIGH_THRESHOLD`, target = `composition.clutter_score * CLUTTER_REDUCTION_FACTOR`, else unchanged |
| `layout_direction.target_negative_space_ratio` | Rule: if below `MIN_NEGATIVE_SPACE_RATIO`, target = the threshold itself |
| `layout_direction.focal_zone` | The largest face's `bbox` if present, else the highest-confidence object's `bbox`, else `None` |
| `subject_treatment.target_bbox`, `target_position_label` | Direct copy from `faces.faces[largest]` / `composition.subject_placement`, with a rule-of-thirds nudge if `composition.rule_of_thirds_score` is below threshold |
| `subject_treatment.crop_tighter` | Rule: `True` if the subject's bbox area is below `MIN_SUBJECT_AREA_RATIO` |
| `text_overlay.avoid_zones` | Computed geometrically as the subject's `bbox` (never place text over the subject) |
| `text_overlay.include_text` | Rule: `True` only if `ocr.text_regions` is non-empty (preserve/relocate existing text) or a specific `elements_to_preserve` entry references text — **never** invents text where none existed |
| `color_direction.target_brightness/contrast/saturation` | Rule: delta math against `BRIGHTNESS_TARGET_RANGE` / `CONTRAST_TARGET_RANGE` / `SATURATION_TARGET_RANGE` constants |
| `color_direction.warm_or_cool` | Pass-through of `colors.warm_or_cool`, unless `GeminiReasoning.weaknesses` explicitly flags color temperature (string-matched against a small fixed keyword set — a rule, not a generative step) |
| `object_directives` | Rule: objects below `YOLO_MIN_CONFIDENCE` already excluded by Module 4; objects contributing most to `composition.clutter_score` when clutter is high are flagged `"remove"`, others `"preserve"` |
| `elements_to_preserve` | **Direct pass-through** from `GeminiReasoning.elements_to_preserve` — no transformation at all |
| `overall_rationale` | Deterministic string template referencing `ctr_potential_score`, `curiosity_gap_score`, and which rules fired — not free generation |

Nothing in this table requires judgment; it requires reading numbers Module 4 already produced and comparing them to constants. That is a rule engine, not a reasoning problem.

## 3. Inputs

Unchanged from v1: Module 5's public entry point takes exactly one required argument, `ThumbnailIntelligence`, and nothing else (no `VideoMetadata`, no `Creator`).

**Guard condition (unchanged in spirit, simplified in cause):** if `thumbnail_intelligence.reasoning is None` (Module 4's Ollama stage failed and degraded) **or** `thumbnail_intelligence.status == "error"`, Module 5 raises `InvalidIntelligenceError` immediately — there is no `ctr_potential_score`/`curiosity_gap_score` to build `overall_rationale` from, and no `elements_to_preserve` to pass through. This is now Module 5's **only** failure mode driven by its input; it has no failure mode driven by an external dependency, because it has no external dependency.

## 4. Output: the `RedesignSpecification` model family

Same conventions as v1 (`BaseModel` + `ConfigDict(frozen=True)`, `Literal` for closed enums, reuse of `BoundingBox` from Module 4), with the generative fields removed.

```python
class ColorDirection(BaseModel):
    """Deterministic target color adjustment for the redesigned thumbnail.
    Contains no invented colors — only directional/magnitude targets
    computed from ColorProfile."""
    model_config = ConfigDict(frozen=True)

    target_brightness: float = 0.5          # [0.0, 1.0], derived from colors.brightness + threshold rule
    target_contrast: float = 0.5            # [0.0, 1.0], derived from colors.contrast + threshold rule
    target_saturation: float = 0.5          # [0.0, 1.0], derived from colors.saturation + threshold rule
    warm_or_cool: Literal["warm", "cool", "neutral"] = "neutral"
    rationale: str = ""                     # deterministic template string, not free text


class SubjectTreatment(BaseModel):
    model_config = ConfigDict(frozen=True)

    has_subject: bool = False
    target_bbox: Optional[BoundingBox] = None
    target_position_label: str = "center"
    crop_tighter: bool = False
    rationale: str = ""


class TextOverlaySpec(BaseModel):
    """Placement-only. Never invents new on-thumbnail copy — see §4.1."""
    model_config = ConfigDict(frozen=True)

    include_text: bool = False
    placement_zone: Optional[BoundingBox] = None
    avoid_zones: list[BoundingBox] = []
    rationale: str = ""


class ObjectDirective(BaseModel):
    model_config = ConfigDict(frozen=True)

    label: str
    action: Literal["include", "remove", "preserve"]
    rationale: str = ""


class LayoutDirection(BaseModel):
    model_config = ConfigDict(frozen=True)

    target_negative_space_ratio: float = 0.0
    target_clutter_score: float = 0.0
    focal_zone: Optional[BoundingBox] = None
    rationale: str = ""


class RedesignSpecification(BaseModel):
    """
    Output of Module 5.

    A fully structured, execution-ready description of what the
    redesigned thumbnail should look like, derived deterministically
    from Module 4's ThumbnailIntelligence via threshold rules and
    arithmetic. Contains no invented creative content (no new hex
    palette, no new on-thumbnail copy) and involves no network call —
    Module 5 makes no AI/LLM call of its own. Module 4 remains the
    only reasoning stage in the pipeline; this model only restructures
    what Module 4 and its own CV stages already measured.
    """
    model_config = ConfigDict(frozen=True)

    video_id: str
    source_thumbnail_path: str               # provenance only; never re-opened

    color_direction: ColorDirection
    subject_treatment: SubjectTreatment
    text_overlay: TextOverlaySpec
    layout_direction: LayoutDirection
    object_directives: list[ObjectDirective] = []

    elements_to_preserve: list[str] = []     # pass-through from GeminiReasoning, unchanged
    overall_rationale: str = ""              # deterministic template, ties back to ctr/curiosity/mismatch

    source_ctr_potential_score: float
    source_curiosity_gap_score: float
    source_content_mismatch_detected: bool

    status: Literal["success", "error"] = "success"
    error_message: Optional[str] = None
    duration_seconds: float = 0.0
    generated_at: str                        # ISO-8601 UTC, same convention as analyzed_at

    @field_validator("video_id")
    @classmethod
    def video_id_must_not_be_empty(cls, v: str) -> str:
        if not v or not v.strip():
            raise ValueError("video_id must not be empty")
        return v.strip()
```

### 4.1 Fields removed, and where they actually belong

| Removed field | Why it's gone | Where it belongs instead |
|---|---|---|
| `ColorDirection.target_palette` (specific new hex values) | Picking genuinely new colors is a creative/generative act, not a rule; keeping it would have required either a hallucination-prone LLM call or a crude, over-engineered color-wheel algorithm masquerading as "deterministic" | Deferred. If truly needed later, a simple, explicitly-labeled deterministic hue-rotation utility could live in Module 6 (Prompt Compiler) as an optional pre-processing step — not in Module 5, and not framed as AI-derived |
| `TextOverlaySpec.suggested_text` (new on-thumbnail copy) | Inventing punchy new copy is unambiguously generative; there is no rule-based path from "reduce clutter" to a specific new headline | This is copywriting, i.e. Module 9's territory (Outreach Copywriter) or a small dedicated copy-generation module — explicitly **not** Module 5 |
| `style_descriptors: list[str]` (freeform mood/style keywords) | These were always going to be LLM-generated adjectives ("dramatic lighting," "high-contrast"); no deterministic source produces them | Same as above — pushed to whichever later module actually needs backend-specific prompt language (Module 6), where it can be scoped as an explicit, separately-reviewed generative step rather than smuggled into Module 5 |
| `partial` status tier | Was reserved for future independent degradation of multiple LLM sub-calls; Module 5 has no LLM calls to degrade | Removed; `status` is now a plain two-value `Literal["success", "error"]` |

This is a deliberate, visible narrowing of Module 5's scope — not a deferral of the problem, a decision that these three concerns are a **different module's responsibility**, one that should be built and reviewed as an explicit generative step later, not folded quietly into what is otherwise a pure-Python transform.

## 5. Configuration additions (`config.py`)

No Ollama-related constants of any kind. Only deterministic threshold/rule constants, following the exact style of the existing Module 4 thresholds (`OCR_MIN_CONFIDENCE`, `FACE_MIN_CONFIDENCE`, `YOLO_MIN_CONFIDENCE`):

```python
# ---------------------------------------------------------------------------
# Module 5 — Redesign Specification Engine (fully deterministic — no
# network calls, no AI/LLM dependency of any kind)
# ---------------------------------------------------------------------------

#: Log file used by Module 5.
MODULE5_LOG_PATH: Path = LOG_DIR / "module5.log"

#: Directory where structured redesign specifications are stored as JSON.
DEFAULT_REDESIGN_SPEC_DIR: Path = PROJECT_ROOT / "data" / "redesign_specs"

#: Filename template for a saved redesign specification; formatted with
#: ``video_id``.
REDESIGN_SPEC_FILENAME_TEMPLATE: str = "{video_id}.json"

# --- Composition thresholds ---

#: Above this, composition.clutter_score is considered high enough to
#: warrant a reduction target.
CLUTTER_HIGH_THRESHOLD: float = 0.6

#: Multiplier applied to clutter_score to compute a reduction target
#: when CLUTTER_HIGH_THRESHOLD is exceeded.
CLUTTER_REDUCTION_FACTOR: float = 0.7

#: Minimum acceptable negative_space_ratio; below this, it becomes the
#: target itself.
MIN_NEGATIVE_SPACE_RATIO: float = 0.25

#: Below this, rule_of_thirds_score is considered poor enough to nudge
#: target_bbox toward the nearest rule-of-thirds intersection.
RULE_OF_THIRDS_LOW_THRESHOLD: float = 0.4

#: Below this fractional bbox area, a subject is considered too small
#: and crop_tighter is set True.
MIN_SUBJECT_AREA_RATIO: float = 0.15

# --- Color thresholds (each a (low, high) acceptable band; values
#     outside the band produce a target at the nearer band edge) ---

BRIGHTNESS_TARGET_RANGE: tuple[float, float] = (0.35, 0.75)
CONTRAST_TARGET_RANGE: tuple[float, float] = (0.4, 0.8)
SATURATION_TARGET_RANGE: tuple[float, float] = (0.3, 0.7)

# --- Weakness-keyword matching (rule-based, not generative) ---

#: Fixed keyword set checked (case-insensitive substring match) against
#: GeminiReasoning.weaknesses to decide whether warm_or_cool should be
#: flipped from the source ColorProfile's reading. Not an LLM call —
#: a small, fully enumerable, testable string-matching rule.
COLOR_TEMPERATURE_FLIP_KEYWORDS: frozenset[str] = frozenset(
    {"too warm", "too cool", "color temperature", "washed out"}
)
```

Nothing from the Module 4 `OLLAMA_*` block is imported or referenced anywhere in Module 5.

## 6. Module structure: `modules/redesign_spec_engine.py`

```
"""Module docstring: responsibility / out-of-scope / storage / public API.
   Explicitly states: fully deterministic, zero network calls, no AI/LLM
   dependency — Module 4 remains the pipeline's only reasoning stage."""

imports:
    from __future__ import annotations
    import json, sys, time
    from datetime import datetime, timezone
    from pathlib import Path
    from typing import Optional
    from loguru import logger
    # NOTE: deliberately absent — requests, tenacity, RetryCallState,
    # retry, retry_if_exception_type, stop_after_attempt, wait_exponential.
    # No network library of any kind is imported.

sys.path bootstrap (same _MODULES_DIR pattern as every other module)
config + models imports (only the deterministic threshold constants
    from §5, plus BoundingBox / GeminiReasoning / ThumbnailIntelligence /
    the new RedesignSpecification family from models.py)

                       # enqueue=True — identical pattern to every other
                       # module; logging is retained even though the
                       # module has no network dependency, because
                       # observability of a rule engine's decisions still
                       # matters (which thresholds fired, for which video_id)

# Custom exceptions — reduced from v1

                                                          # reasoning is None,

                                                          # on read/write only
# REMOVED from v1: _RedesignTransientError, RedesignReasoningError —
# both existed only to model network/schema failure, which can no
# longer occur.

# --- Pure derivation functions (the actual rule engine) ---
# Each is a small, independently unit-testable, side-effect-free function
# operating only on already-validated Pydantic model fields + config

def _build_overall_rationale(reasoning: GeminiReasoning, fired_rules: list[str]) -> str: ...
    # String template, e.g.:
    # "ctr_potential_score={score:.2f}; rules applied: {', '.join(fired_rules)}"
    # Never calls any generation step — pure string formatting.

# --- Orchestration ---

def build_redesign_specification(
    intelligence: ThumbnailIntelligence,
) -> RedesignSpecification:
    """
    Deterministically derive a RedesignSpecification from
    ThumbnailIntelligence. No parameters beyond `intelligence` — unlike
    Module 4's analyze_thumbnail, there is no injectable reasoning
    function, because there is nothing to inject: no external call is
    made.

    Raises:
        InvalidIntelligenceError: if intelligence.reasoning is None or
            intelligence.status == "error".
    """
    # 1. Validate input usability (raise InvalidIntelligenceError early).
    # 2. Call each _derive_* function in sequence, timing the whole
    #    operation (time.monotonic() start/end -> duration_seconds).
    # 3. Assemble RedesignSpecification with status="success".
    # 4. Log completion at INFO, matching Module 4's per-video_id logging
    #    style.
    # There is no partial/degraded path: each _derive_* function is a
    # total function over its (already-validated) inputs and cannot
    # itself fail in a way that warrants degrading to a safe default —
    # if it raises, that is a bug, not an expected runtime condition,
    # and should propagate as an uncaught error during development/tests
    # rather than being silently swallowed.

# --- JSON persistence (unchanged pattern from Module 3/4) ---

def save_redesign_spec(spec: RedesignSpecification, spec_dir=DEFAULT_REDESIGN_SPEC_DIR) -> None: ...

    # Identical "corrupted cache -> None, log warning" pattern as
```

### 6.1 What's explicitly removed from the v1 module structure

- `_call_ollama_api()` — deleted entirely. No replacement; there is no network call in Module 5.
- All `tenacity` imports and usage — `@retry`, `stop_after_attempt`, `wait_exponential`, `retry_if_exception_type`, `RetryCallState`, `_before_sleep_log_redesign` — deleted entirely. There is nothing transient to retry.
- `_extract_json_object()` / `_strip_markdown_fences()` — deleted entirely (and, per the v1 open question, **not** extracted into a shared helper module either, since neither Module 4 nor Module 5 needs to share them once Module 5 has no LLM response to parse). **This resolves v1's open question in the simplest possible direction: Module 4 stays completely untouched, not even a housekeeping import move.**
- The Ollama system-prompt constant and all prompt-engineering text — deleted entirely. There is no prompt in this module.
- `RedesignReasoningError`, `_RedesignTransientError` — deleted; no failure mode they described can occur anymore.
- The `generate_redesign_fn=None` injectable-override parameter on the orchestration function — deleted; it existed solely to let tests mock the LLM call. With no LLM call, there's nothing to mock.
- The `"partial"` status value — deleted from the `Literal`; retained only conceptually in §4.1's removed-fields table as "no longer applicable," not as a reserved-for-later value, since there's no longer a plausible future path to partial LLM degradation once Module 5 has no LLM at all.

## 7. Testing strategy

This is the area that changes most substantively, because removing the network dependency removes an entire category of test.

**Removed test categories (relative to what Module 4's suite needed, and what v1's Module 5 would have needed):**
- No `monkeypatch.setitem(sys.modules, "requests", ...)` or equivalent network-call faking.
- No `TestCallOllamaApi`-equivalent class.
- No `TestOllamaLiveIntegration`-equivalent class (there is no live service to integration-test against).
- No tests for retry/backoff behavior, timeout handling, or malformed-JSON recovery — none of that machinery exists to test.
- No golden-prompt or prompt-regression tests.

**What the suite looks like instead** — structured the same way as the existing test files (`TestXxx` classes per function, `tmp_path` for filesystem isolation), but every test is a pure unit test with fully deterministic, hand-constructed inputs and exact expected outputs — no mocking required at all, because there is nothing external to mock:

```
class TestDeriveColorDirection:
    # Table-driven: for each (brightness, contrast, saturation, weaknesses)
    # input tuple, assert the exact expected ColorDirection output.
    # Covers: in-band values pass through unchanged; below-band and
    # above-band values clamp to the nearer threshold edge; each
    # COLOR_TEMPERATURE_FLIP_KEYWORDS keyword correctly flips warm_or_cool;
    # absent keywords leave it unchanged.

class TestDeriveSubjectTreatment:
    # Covers: no faces -> has_subject=False, target_bbox=None; single
    # face above/below MIN_SUBJECT_AREA_RATIO -> crop_tighter True/False;
    # multiple faces -> largest face selected; rule_of_thirds_score below
    # threshold -> bbox nudged toward nearest intersection (exact expected
    # coordinates asserted).

class TestDeriveTextOverlay:
    # Covers: empty ocr.text_regions -> include_text=False; non-empty
    # text_regions -> include_text=True and avoid_zones exactly equals
    # the subject bbox; no subject -> avoid_zones empty.

class TestDeriveLayoutDirection:
    # Covers: clutter_score above/below CLUTTER_HIGH_THRESHOLD; 
    # negative_space_ratio above/below MIN_NEGATIVE_SPACE_RATIO;
    # focal_zone selection priority (largest face over largest object,
    # None when neither present).

class TestDeriveObjectDirectives:
    # Covers: empty objects -> empty directives; objects present with
    # clutter_score above/below threshold -> "remove" vs "preserve"
    # assigned correctly, ordered by confidence/size as documented.

class TestBuildOverallRationale:
    # Covers: exact string template output for a given set of fired
    # rules and scores — a pure string-formatting assertion.

class TestBuildRedesignSpecification:
    # Orchestration-level tests using fully hand-built ThumbnailIntelligence
    # fixtures (success, reasoning=None, status="error") — asserting
    # InvalidIntelligenceError is raised in the latter two cases and a
    # fully-populated RedesignSpecification in the first. No injectable
    # override needed since there is no external call to substitute.

class TestSaveRedesignSpec:
    # Identical pattern to TestSaveIntelligence — tmp_path, atomic write,
    # OSError -> RedesignSpecCacheError.

class TestLoadCachedRedesignSpec:
    # Identical pattern to TestLoadCachedIntelligence — cache hit, cache
    # miss, corrupted JSON -> None + logged warning.
```

Every test in this suite runs in well under a millisecond and needs no `monkeypatch` beyond the filesystem-error-injection pattern already used in `test_csv_reader.py`'s `_RaisingCtx` idiom for the cache-error tests. This is a materially smaller, faster, and more exhaustively coverable test surface than v1's design would have required — the entire suite can run to 100% branch coverage deterministically, which was not achievable for the LLM-calling version without either hitting a real Ollama instance or accepting untested prompt/parsing paths.

## 8. Storage & downstream consumption (updated)

**Storage:** unchanged — `data/redesign_specs/{video_id}.json`.

**Downstream consumption — updated to reflect where the removed fields now live:**

- **Module 6 (Prompt Compiler)** reads `RedesignSpecification` and compiles it into a specific image-generation backend's prompt/parameter format. **This is now also where any deterministic color-wheel hue-rotation utility would live**, if a concrete new palette is ever needed beyond the brightness/contrast/saturation targets Module 5 already provides — kept out of Module 5, as decided in §4.1.
- **Module 7 (Image Generation Engine)** — unchanged from v1: the only module that generates pixels.
- **Module 8 (Redesign QA)** — unchanged from v1: reuses Module 4's existing public CV functions as a library against the newly generated image.
- **Module 9 (Outreach Copywriter)** — **now also owns on-thumbnail copy suggestion** (formerly `TextOverlaySpec.suggested_text` in v1), in addition to the outreach email body. This consolidates all generative-copy responsibility for a given creator into one module, joined against `VideoMetadata`/`Creator` via `video_id`, rather than splitting "invent thumbnail text" and "invent email text" across two different modules.
- **Module 10 (Email Assembler)**, **Module 11 (Gmail Sender)**, and the `main.py` pipeline extension — unchanged from v1.

## 9. Consistency checklist against existing architecture (updated)

| Concern | Module 4's approach | Module 5's approach (v2) |
|---|---|---|
| Config | Centralized constants, per-module section | Same — new section, but every constant is a deterministic threshold/rule value; zero `OLLAMA_*` references |
| Logging | `loguru`, idempotent `_configure_logger()`, rotating sink | Identical — retained for observability of rule firing, despite no network dependency |
| Retries | `tenacity`, exponential backoff on transient-marker exceptions | **Not applicable — removed entirely.** Nothing in Module 5 is retryable because nothing is transient; every operation is a pure function over already-validated data |
| Network calls | One deliberate local Ollama call, isolated to a single function (`_call_ollama_api`) | **None.** Zero network imports, zero network calls, anywhere in the module |
| Caching | Cache-check helper, corrupted cache → warning + miss, no auto-consult inside main function | Identical pattern retained for `load_cached_redesign_spec` / `save_redesign_spec` — this part of the architecture is orthogonal to whether the *generation* step is deterministic or not |
| Validation | Pydantic, `frozen=True`, `field_validator` | Identical |
| Error handling | Per-module exception hierarchy | Narrower hierarchy: only `InvalidIntelligenceError` (bad input) and `RedesignSpecCacheError` (filesystem I/O) — no network/parsing-failure exceptions exist to model |
| Testing | `pytest`, `tmp_path`, `monkeypatch.setitem` for faking heavy deps, injectable override for mocking the LLM call | Pure unit tests only; no mocking of any kind is required for the derivation logic, since there is no external dependency to substitute |
| JSON persistence | Atomic tmp-write + `Path.replace` | Identical |

## 10. Why this shape fits the repository, specifically

1. **It restores the line the codebase already draws for itself.** Modules 1–3, and five of Module 4's six internal stages, are deterministic; Ollama is reached for exactly once, for the one genuinely irreducible judgment call in the entire pipeline (CTR/curiosity potential, which depends on nothing computable). A second reasoning stage in Module 5 would have broken that principle for no corresponding gain, since everything Module 5 needs is already numeric.
2. **It's directly supported by evidence already in the repo, not just a stylistic preference.** `ollama_response.txt` shows the existing, smaller schema already fails to parse sometimes. Adding a second, larger-schema LLM dependency downstream would compound a reliability problem the project is *already* experiencing, not a hypothetical one.
3. **It keeps the "no invented content" boundary structural rather than promised.** By deleting `target_palette` and `suggested_text` from the model entirely — rather than leaving them present-but-unused — there is no field in `RedesignSpecification` a future maintainer could accidentally start populating with an LLM call without first reopening this document and adding the field back.
4. **It makes the module trivially, exhaustively testable**, matching Module 1's "full test coverage" bar in practice rather than aspiration: every branch of every rule can be asserted directly, with no live-service dependency and no flaky-mock risk.
5. **It's a strict simplification, not a compromise.** Nothing Module 5 needs to do is left undone — every quantitative field in the original spec is still produced, just via arithmetic instead of a network round-trip. The only things actually cut are the two fields that were never really "redesign specification" in nature to begin with; they've been relocated to modules whose job is explicitly generative (Module 9), where that responsibility can be reviewed and tested on its own terms.

## 11. Open questions carried forward (updated)

- **v1's open question about extracting `_extract_json_object`/`_strip_markdown_fences` into a shared helper is now moot** — Module 5 no longer parses any LLM response, so there's nothing to share. Module 4 stays completely untouched.
- **Rule constants in §5 (`CLUTTER_HIGH_THRESHOLD`, `BRIGHTNESS_TARGET_RANGE`, etc.) are initial proposals**, not empirically tuned. Worth reviewing against a sample of real `ThumbnailIntelligence` outputs before finalizing, the same way `OCR_MIN_CONFIDENCE`/`FACE_MIN_CONFIDENCE` presumably were.

If it feels like scope creep even at this scale, the simplest fix is to drop it and have `warm_or_cool` pass through `colors.warm_or_cool` unconditionally, ignoring `weaknesses` entirely for this one field.
